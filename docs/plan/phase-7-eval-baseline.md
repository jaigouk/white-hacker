# Phase 7 — Eval baseline (a labeled finding set; FP-rate tracking)

> **Theme:** stand up the *minimum* eval baseline the inner loop needs to validate releases: a labeled
> finding set (planted vulns + clean look-alikes), a deterministic scorer, and an FP-rate number tracked
> before each release. This is the **seed** that Phase 9 freezes into the keep-or-revert corpus the outer
> loop depends on.
> **Maps to:** PLAN §8.1 P7; si-08 §6 (corpus design — the *baseline* subset), §6.2 (Youden's J).
>
> **Loop position:** BRIDGE. Phase 7 = "does this release regress against a labeled set?"; Phase 9 hardens
> the same corpus into the frozen, agent-read-only gate. Build Phase 7 first; Phase 9 freezes it.
> **Exit condition:** `score.py` runs the pipeline over the labeled set and prints TPR/FPR + Youden's J;
> a release checklist records the FP rate; the corpus reuses the Phase-0/4/5 fixtures already on disk.

> **Relationship to Phase 9.** Phase 7 is the *working* eval (the agent may add cases as it builds
> fixtures). Phase 9 **freezes** it: marks it read-only to the agent (hook-enforced), adds the asymmetric
> keep-or-revert thresholds, and wires it as a merge gate for any `sec-learn`/`sec-kb-refresh` change.
> Do not let any outer-loop self-write merge until Phase 9 is done (README ordering note).

## Grooming (re-groomed 2026-06-06, after Phase 6)

**Readiness:** ✅ READY. Phases 0–6 done. Seeds exist (`poc-floor-review` 3 vuln/clean pairs;
`poc-ai-review` vuln + clean look-alike; `poc-patch-ladder` command-injection).

**Reconciliations (own these):**
1. **`score.py` is a deterministic SCORER, not an agent-runner.** It takes a **findings JSON**
   (finding-schema) + the corpus **labels** and computes TPR / FPR / **Youden's J**. The *pipeline run*
   that produces findings is the agent (`/security-review`, non-deterministic) — out of scope for the
   unit tests. So `score.py` is tested on **synthetic** findings-vs-labels with known TP/FP/FN
   (a perfect run → J=1; an all-FP run → J≤0), which is fully deterministic.
2. **Matching rule (avoid line brittleness):** a finding matches a label when `file` matches AND
   `category` matches AND `line` is within a **±tolerance** (default ±3) — tested with a tolerance edge
   case. A label's `benign_lookalike` must produce **no** finding (the FP term).
3. **Label schema is new** → define `evals/label-schema.json` (Draft 2020-12): per case
   `{case_id, language, category, severity, vulnerable:{file,line}, benign_lookalike:{file}}`. The
   pairing test asserts every case has BOTH variants on disk.
4. **Corpus ≥30 (target ≥100 in Phase 9):** seed the ~5 existing pairs, then generate the rest across
   the `core-checklist.md` categories (SQL/command/path/SSTI/LDAP/NoSQL injection; BOLA/BFLA/JWT authz;
   SSRF; weak-crypto/hardcoded-secret; pickle/yaml deserialization; XSS; CORS/config) **and** the AI
   classes (prompt-injection, LLM05 improper-output, excessive-agency) across py/js/go. Cases are
   **small** (a few lines each), vulnerable + clean look-alike + `label.json`.
5. **`baseline.json` is reproducible:** record `score.py` run over a **checked-in findings snapshot**
   (`evals/runs/baseline-findings.json`) → `{tpr,fpr,youden_j,date,commit}`. Re-running `score.py` on
   the same snapshot+labels reproduces the number exactly (determinism = same inputs, no agent call).
   When a real agent run is captured later it replaces the snapshot.
6. **Phase-9 boundary:** Phase 7 is the *working* eval (agent may add cases); Phase 9 freezes it
   (read-only hook, asymmetric keep-or-revert thresholds, merge gate). Don't build the freeze here.

**Generation note:** the bulk corpus cases are generated by a fan-out workflow whose agents **return
case content as data only (no file writes, no `git`)**; the parent writes the files and validates each
`label.json` against `label-schema.json` (avoids the stray-edit/commit side-effects seen in Phase 5).

**Order:** `label-schema.json` + `score.py` (+ tests) → corpus (T-7.1) → `baseline.json` + checklist.

**DoD (phase):** `evals/corpus/cases/` has ≥30 paired cases, every `label.json` schema-valid, AI sinks
represented; `score.py` returns arithmetically-correct TPR/FPR/J on synthetic sets (machine JSON,
tolerance-matched); `baseline.json` validates + reproduces on re-run; `docs/release-checklist.md` has
the "run score.py, compare to baseline" step. Then **re-groom Phase 8**.

### T-7.2 · Deterministic scorer (`score.py`, Youden's J)
- **Goal:** a scorer that runs the review pipeline over the corpus, compares against labels, and emits
  TPR, FPR, and **Youden's J (TPR − FPR)** (si-08 §6.2 — the OWASP Benchmark convention) as a single
  deterministic number per run, plus per-category breakdown.
- **Artifact:** `evals/score.py` (+ `evals/pyproject.toml`, `evals/tests/test_score.py`)
- **Depends on:** T-7.1, T-1.1
- **Verification criteria:**
  - [x] On synthetic sets with known TP/FP/FN, `score.py` returns correct TPR/FPR/J — `uv run --with jsonschema --with pytest pytest evals/tests/test_score.py` *(9 tests: perfect run J=1, all-FP J=-1, no-findings J=0)*
  - [x] Output is machine-readable JSON (`{tpr, fpr, youden_j, …, by_category}`) — `test_output_shape_and_by_category`
  - [x] Matches by `file` + category + line-within-tolerance (no exact-line brittleness) — `test_line_tolerance` (±3); case-qualified to avoid basename collisions across cases
- **Status:** done
